# Middleware - FastAPI Documentation

## Overview

A "middleware" is a function that works with every **request** before it is processed by any specific path operation, and also with every **response** before returning it.

### How Middleware Works

- It takes each **request** that comes to your application
- It can perform operations on that request or run needed code
- Then it passes the request to be processed by the rest of the application (by some path operation)
- It then takes the **response** generated by the application
- It can perform operations on that response or run needed code
- Then it returns the response

**Technical Details**: If you have dependencies with `yield`, the exit code will run *after* the middleware. If there are background tasks, they will run *after* all the middleware.

## Create a Middleware

To create a middleware, use the `@app.middleware("http")` decorator on a function.

The middleware function receives:
- The `request`
- A function `call_next` that will receive the `request` as a parameter
  - This function passes the request to the corresponding path operation
  - Then it returns the response generated by that path operation
- You can then further modify the response before returning it

### Example: Adding Process Time Header

```python
import time
from fastapi import FastAPI, Request

app = FastAPI()

@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    start_time = time.perf_counter()
    response = await call_next(request)
    process_time = time.perf_counter() - start_time
    response.headers["X-Process-Time"] = str(process_time)
    return response
```

**Tip**: Use `time.perf_counter()` instead of `time.time()` because it can be more precise for these use cases.

**Custom Headers**: Custom proprietary headers can be added using the `X-` prefix. If you want custom headers to be visible to clients in a browser, you need to add them to your CORS configurations using the `expose_headers` parameter.

## Multiple Middleware Execution Order

When you add multiple middlewares, each new middleware wraps the application, forming a stack. The last middleware added is the *outermost*, and the first is the *innermost*.

### Example

```python
app.add_middleware(MiddlewareA)
app.add_middleware(MiddlewareB)
```

Results in the following execution order:

- **Request path**: MiddlewareB -> MiddlewareA -> route
- **Response path**: route -> MiddlewareA -> MiddlewareB

This stacking behavior ensures that middlewares are executed in a predictable and controllable order.
